# 2. 參數一致性

前端儲存 API 的 request 格式中參數名稱要與查詢 API 拿到的參數一致，這樣可以提升串接效率，減少不必要的欄位轉換。

---

## ✓ 正確範例

### 查詢 API 回傳格式
若使用駝峰命名，大小寫需一致（含縮寫字母大小寫）。
```json
{
  "userId": 123,
  "userName": "Jane",
  "createdAt": "2026-02-12T10:30:00"
}
```

### 新增/編輯 API request 格式
使用駝峰命名，大小寫需一致（含縮寫字母大小寫）。
```json
{
  "userId": 123,
  "userName": "Jane",
  "createdAt": "2026-02-12T10:30:00"
}
```

**優點：**
- 參數名稱完全一致，可直接使用查詢結果
- 不需要額外的欄位名稱轉換邏輯
- 降低串接錯誤率，提升開發效率

---

## ✗ 錯誤範例

### 查詢 API 回傳格式
```json
{
  "name": "Jane",
  "age": 25,
  "email": "jane@example.com"
}
```

### 新增/編輯 API request 格式
```json
{
  "userName": "Jane",
  "userAge": 25,
  "userEmail": "jane@example.com"
}
```

**問題：**
- 參數名稱不一致（name vs userName、age vs userAge）
- 需要額外撰寫欄位轉換邏輯
- 容易因遺漏轉換導致串接錯誤
- 增加維護成本與測試複雜度

---

## 為何要做到一致性？
- **簡化前端開發流程：** 查詢後可直接將資料綁定到表單，不需額外轉換欄位名稱
- **降低出錯機率：** 避免因欄位名稱對應錯誤導致的 bug
- **提升可維護性：** 減少欄位映射邏輯，程式碼更簡潔易懂
- **加速開發效率：** 前後端協作更順暢，減少溝通成本

---

## 什麼情況下可以做到一致性？
- **新開發的功能：** 在設計階段就統一前後端欄位名稱規範
- **重構既有功能：** 在大規模重構時，同步調整前後端欄位命名
- **後端使用 DTO 轉換：** 後端可透過 Data Transfer Object 統一查詢與儲存的欄位結構
- **前後端協作良好：** 團隊有共識且願意遵循統一的命名規範

> ⚠️ **例外情況：** 當頁面邏輯較複雜，或需整合多支查詢 API 的資料時，可不強制對齊參數名稱。
