# 4. 分頁資料格式

分頁功能採用關注點分離設計，分為 usePagination（邏輯層）和 AppPagination（展示層）兩個部分。

---

## 🤝 前後端協作規範

### 📌 Request 參數組成
- **固定參數**：pageNo、pageSize（所有分頁 API 都需要）
- **業務參數**：typeId、keyword、startDate 等（視 API 功能需求而定，非必要）

### 情境 A：只需分頁
```
POST /api/users/list

{
  "pageNo": 1,
  "pageSize": 15
}
```
例如：取得所有使用者列表

### 情境 B：需查詢條件 + 分頁
```
POST /api/news/query

{
  "typeId": 4,
  "keyword": "測試",
  "pageNo": 1,
  "pageSize": 15
}
```
例如：根據條件搜尋新聞

### Request 參數說明

| 參數 | 型別 | 必填 | 說明 | 預設值 |
|---|---|:---:|---|---|
| `pageNo` | Integer | ✅ | 頁碼（1-based，第一頁 = 1） | 1 |
| `pageSize` | Integer | ✅ | 每頁筆數 | 15 |
| `typeId` | Integer | ❌ | 類別 ID（範例） | - |
| `keyword` | String | ❌ | 搜尋關鍵字（範例） | - |
| `startDate` | String | ❌ | 開始日期（範例） | - |

### 回應格式
```json
{
  "content": [...],
  "number": 1,
  "size": 15,
  "totalPages": 10,
  "total": 150,
  "offset": 0
}
```

### Response 參數說明

| 參數 | 型別 | 必填 | 說明 |
|---|---|:---:|---|
| `content` | Array | ✅ | 資料陣列（空時為 []，不可為 null） |
| `number` | Integer | ✅ | 當前頁碼（1-based，對應 request.pageNo） |
| `size` | Integer | ✅ | 每頁筆數（對應 request.pageSize） |
| `totalPages` | Integer | ✅ | 總頁數 |
| `total` | Integer/Long | ✅ | 總筆數 |
| `offset` | Integer/Long | ✅ | 起始索引 = (number - 1) × size |

## 💻 前端實作指引

### 📐 架構概述
分頁功能採用 **關注點分離（Separation of Concerns）** 設計模式，分為兩個層次：
- **usePagination.js**（邏輯層）：管理分頁狀態、處理 API 參數轉換、解析 API 回應
- **AppPagination.vue**（展示層）：渲染分頁按鈕、處理使用者點擊、發送 update:page 事件

---

### 🔧 usePagination Composable

**檔案位置：** `src/composables/usePagination.js`
**功能：** 提供分頁狀態管理和資料處理的可組合函數

#### 回傳值
```javascript
{
  paginationModel,           // 分頁狀態物件
  updatePaginationPage,      // 更新頁碼函數（必須使用此函數處理換頁，才會觸發 API 請求）
  mergePaginationParameter,  // 合併分頁參數到 API 請求
  extractPaginationContent,  // 從 API 回應中提取分頁資料
}
```

#### paginationModel 結構
```javascript
{
  page: 1,           // 當前頁碼（起始值：1）
  pageSize: 15,      // 每頁筆數（預設：15）
  itemCount: 0,      // 總筆數
  pageCount: 0,      // 總頁數
  prefix: Function,  // 分頁前綴文字生成函數
}
```

---

### 📋 主要方法

#### 1. mergePaginationParameter(parameter)
將分頁參數合併到 API 請求參數中。

> **內部機制：** `pageNo` 來自內部的 `requestPage`（而非 `paginationModel.page`），只有透過 `updatePaginationPage` 才會更新 `requestPage`。這確保 `extractPaginationContent` 回寫 `paginationModel.page` 時不會觸發 computed 重算，避免無限 API 循環。

**輸入：**
```javascript
{ keyword: '測試' }
```

**輸出：**
```javascript
{
  keyword: '測試',
  pageNo: 1,      // 頁碼（從 1 開始）
  pageSize: 15    // 每頁筆數
}
```

#### 2. extractPaginationContent(response)
從 API 回應中提取資料並更新分頁狀態。

**支援兩種 API 回應格式：**

格式 1（推薦）：使用 page/size
```javascript
{
  content: [...],
  page: 1,
  size: 15,
  totalPages: 10,
  total: 150,
  offset: 0
}
```

格式 2：使用 number/pageSize
```javascript
{
  content: [...],
  number: 1,
  pageSize: 15,
  totalPages: 10,
  total: 150,
  offset: 0
}
```

> **自動處理：** 提取 content 陣列並更新 paginationModel、為每筆資料附加 $$RowNumber（序號）和 $$RowIndex（索引）

---

### ⚠️ 換頁必須使用 updatePaginationPage

> 🚨 **重要行為規則：所有換頁操作一律使用 updatePaginationPage**
> 不能直接賦值 `paginationModel.page = page`，這樣**只會更新 UI，不會觸發 API 請求**。

#### ❌ 錯誤寫法
```javascript
// 直接賦值 → 只更新 UI，不觸發 API
paginationModel.value.page = 2

// 事件處理中直接賦值 → 同樣不會觸發 API
@update:page="page => (paginationModel.page = page)"
```

#### ✅ 正確寫法
```javascript
// 使用函數 → 同時更新 requestPage 和 UI，觸發 API
updatePaginationPage(2)

// 事件處理中使用函數
@update:page="updatePaginationPage"
```

#### 行為對比

| 操作 | 更新 requestPage | 更新 UI (paginationModel.page) | 觸發 API |
|---|:---:|:---:|:---:|
| `paginationModel.page = 2` | ❌ | ✅ | ❌ |
| `updatePaginationPage(2)` | ✅ | ✅ | ✅ |

**為什麼這樣設計？**

`mergePaginationParameter` 產生 `pageNo` 時，讀取的是內部獨立的 `requestPage`，**不是** `paginationModel.page`。

這個設計是為了避免無限循環：`extractPaginationContent` 會回寫 `paginationModel.page`，如果 `mergePaginationParameter` 也讀取它，就會形成 computed → API → 回寫 → computed 的無限循環。透過 `requestPage` 將兩者分離，徹底切斷循環。

```
extractPaginationContent 寫入 paginationModel.page
  → mergePaginationParameter 的 computed 重算
  → 產生新物件觸發 vue-query
  → API 請求 → 回應
  → extractPaginationContent 又寫入
  → 無限循環 ♻️
```

### 🎨 AppPagination 元件

**檔案位置：** `src/components/AppPagination.vue`
**功能：** 專為無障礙網頁設計的分頁 UI 元件，符合 WCAG 2.1 AA 無障礙標準，主要用於 public 頁面（非登入頁面）以通過無障礙檢測

#### Props

| Props | 類型 | 必填 | 預設值 | 說明 |
|---|---|:---:|---|---|
| `page` | Number | ✅ | - | 當前頁碼（1-based） |
| `total` | Number | ✅ | - | 總筆數 |
| `pageSize` | Number | ❌ | 10 | 每頁筆數 |
| `maxPageButtons` | Number | ❌ | 7 | 最多顯示幾個頁碼按鈕 |

#### Events

| Events | 參數 | 說明 |
|---|---|---|
| `update:page` | (page: Number) | 當使用者點擊頁碼時觸發 |

#### ✨ 無障礙特性（WCAG 2.1 AA 標準）

> **重要提醒：** 本元件專為通過政府網站無障礙檢測而設計，確保視障、肢障等使用者也能順利操作分頁功能。

- **ARIA 標籤：** `aria-label` 為每個按鈕提供明確說明（如「第 3 頁」、「上一頁」）
- **當前頁標示：** `aria-current="page"` 告知螢幕閱讀器目前所在頁面
- **鍵盤導航：** 完整支援 Tab 鍵切換、Enter/Space 鍵觸發
- **視覺對比：** 當前頁高亮、disabled 狀態明確，符合色彩對比度要求
- **螢幕閱讀器：** 提供完整的頁碼資訊朗讀（如「共 150 筆資料」）
- **焦點管理：** 清晰的焦點框線，易於追蹤當前位置

#### ⚠️ 重要觀念：Props ≠ API Request 參數

AppPagination 是純 UI 元件，只負責顯示分頁介面，**不負責 API 通訊**。

- **元件 Props（page, pageSize）**：只是 Vue 元件內部的命名，用於接收資料來顯示 UI
- **API Request 參數**：由 usePagination 的 mergePaginationParameter 產生，實際發送的是 `pageNo` 和 `pageSize`

> **簡單來說：** AppPagination 的 `:page` prop 只是傳資料給 UI 元件，與後端 API 參數無關。

### 💡 使用方式

#### 方式一：單獨使用 AppPagination

適用於自訂列表佈局的情況：

```vue
<template>
  <div>
    <!-- 自訂的列表 -->
    <div v-for="item in dataList" :key="item.id">
      {{ item.title }}
    </div>

    <!-- 分頁元件 -->
    <app-pagination
      :page="paginationModel.page"
      :total="paginationModel.itemCount"
      :page-size="paginationModel.pageSize"
      @update:page="updatePaginationPage"
    />
  </div>
</template>

<script setup>
import { useGetDataList } from '@/api/example'

const { paginationModel, updatePaginationPage, mergePaginationParameter, extractPaginationContent } = usePagination()

const queryParams = computed(() => mergePaginationParameter({}))
const { data: apiResponse } = useGetDataList(queryParams)
const dataList = computed(() => extractPaginationContent(apiResponse.value))
</script>
```

#### 方式二：搭配 AppTable 使用 ✅ 推薦

適用於需要表格展示的情況，分頁會自動整合在表格下方：

```vue
<template>
  <app-table
    striped
    single-line
    accessible
    :pagination="paginationModel"
    @update:page="updatePaginationPage"
  >
    <tbody>
      <tr v-for="item in dataList" :key="item.id">
        <td>{{ item.title }}</td>
        <td>{{ item.date }}</td>
      </tr>
    </tbody>
  </app-table>
</template>

<script setup>
import { useGetDataList } from '@/api/example'

const { paginationModel, updatePaginationPage, mergePaginationParameter, extractPaginationContent } = usePagination()

const queryParams = computed(() => mergePaginationParameter({}))
const { data: apiResponse } = useGetDataList(queryParams)
const dataList = computed(() => extractPaginationContent(apiResponse.value))
</script>
```

> ⚠️ **AppTable 重點說明：**
> - 必須加上 `accessible` prop（public 頁面使用無障礙版本）
> - 傳入 `:pagination="paginationModel"`（將整個分頁狀態傳入）
> - 監聽 `@update:page` 事件並呼叫 `updatePaginationPage`
> - **必須使用 `updatePaginationPage` 處理換頁事件，直接修改 `paginationModel.page` 不會觸發 API 請求！**

### 📝 完整範例

#### 搭配搜尋功能（AppTable）

```vue
<template>
  <div>
    <!-- 搜尋表單 -->
    <div class="mb-4">
      <input v-model="searchModel.keyword" placeholder="請輸入關鍵字" />
      <button @click="handleSearch">搜尋</button>
      <button @click="handleReset">清除</button>
    </div>

    <!-- 表格 + 分頁 -->
    <app-table
      striped
      single-line
      accessible
      :pagination="paginationModel"
      @update:page="updatePaginationPage"
    >
      <tbody>
        <tr v-for="item in dataList" :key="item.id">
          <td>{{ item.title }}</td>
          <td>{{ item.date }}</td>
        </tr>
      </tbody>
    </app-table>
  </div>
</template>

<script setup>
import { cloneDeep } from 'lodash-es'
import { useGetNewsList } from '@/api/news'

// 搜尋條件
const searchModel = ref({
  keyword: '',
})

// 實際送出的查詢參數
const queryParams = ref({})

// 分頁管理
const { paginationModel, updatePaginationPage, mergePaginationParameter, extractPaginationContent } = usePagination()

// 合併分頁參數
const apiParams = computed(() => mergePaginationParameter(queryParams.value))

// 呼叫 API
const { data: apiResponse } = useGetNewsList(apiParams)

// 提取資料
const dataList = computed(() => extractPaginationContent(apiResponse.value))

// 執行搜尋
const handleSearch = () => {
  queryParams.value = cloneDeep(searchModel.value)
}

// 清除條件
const handleReset = () => {
  searchModel.value = { keyword: '' }
  queryParams.value = {}
}
</script>
```

### ❓ 常見問題

#### Q1: 應該使用哪種方式？
- **有表格展示** → 使用方式二（AppTable + pagination prop）✅ 推薦
- **自訂列表佈局** → 使用方式一（單獨 AppPagination）

#### Q2: 為什麼 AppTable 的分頁點擊沒反應？
忘記監聽 `@update:page` 事件！必須這樣寫：

```vue
<!-- ❌ 錯誤：沒有監聽事件 -->
<app-table :pagination="paginationModel">

<!-- ❌ 錯誤：直接修改 paginationModel.page，不會觸發 API 請求 -->
<app-table :pagination="paginationModel" @update:page="page => (paginationModel.page = page)">

<!-- ✅ 正確：使用 updatePaginationPage -->
<app-table
  :pagination="paginationModel"
  @update:page="updatePaginationPage"
>
```

**運作原理：**
1. 使用者點擊頁碼按鈕
2. AppPagination 元件發出 `update:page` 事件
3. AppTable 接收到事件後，再次發出 `update:page` 事件
4. 父元件透過 `updatePaginationPage` 同時更新內部 `requestPage` 和 `paginationModel.page`
5. `requestPage` 更新後，`mergePaginationParameter` 的 computed 重新計算，觸發 API 請求

#### Q3: AppTable 的 accessible prop 是什麼？
控制使用哪一種分頁元件：
- `accessible` - 使用 **AppPagination 無障礙版本**（符合 WCAG AA 標準）
- 不加或 `accessible=false` - 使用 Naive UI 的 n-pagination（無障礙不合規）

```vue
<!-- ✅ Public 頁面（對外服務，必須符合無障礙標準） -->
<app-table accessible :pagination="paginationModel">

<!-- ❌ Private 頁面（內部管理系統，不需無障礙檢測） -->
<app-table :pagination="paginationModel">
```

> 📌 **重要提醒：** 所有 **public 資料夾下的頁面**都必須加上 `accessible`，政府網站無障礙檢測會檢查這些頁面。

#### Q4: 為什麼要分成兩個部分（usePagination + AppPagination）？
遵循 Vue 3 Composition API 的最佳實踐：
- **usePagination**：可重複使用的邏輯（純 JavaScript，可用於不同的 UI）
- **AppPagination**：可自訂的展示層（Vue 元件，可替換成其他 UI）

這樣的設計讓你可以：
1. 在不同元件間共用分頁邏輯
2. 輕鬆替換 UI 元件而不影響邏輯
3. 更容易測試和維護

#### Q5: 可以不使用 AppPagination 元件嗎？
可以！`usePagination` 是獨立的，你可以：
1. 使用其他分頁 UI 元件
2. 自己手刻分頁 HTML
3. 使用第三方套件

只要透過 `updatePaginationPage` 更新頁碼即可（不能直接修改 `paginationModel.page`）。

#### Q6: API 回應格式不符合怎麼辦？

> ⚠️ **重要：** 後端應該遵循統一的分頁格式規範。如果發現格式不符合，請**優先回報後端團隊修正**，而不是前端自行轉換。

正常情況下**不需要特殊處理**，因為：
- ✅ **後端應遵循標準格式**：專案後端 API 已統一使用標準分頁格式
- ✅ **前端自動適配**：`extractPaginationContent` 已支援兩種常見格式

格式 1（推薦）：使用 page/size
```json
{
  "content": [...],
  "page": 1,
  "size": 15,
  "totalPages": 10,
  "total": 150
}
```

格式 2：使用 number/pageSize
```json
{
  "content": [...],
  "number": 1,
  "pageSize": 15,
  "totalPages": 10,
  "total": 150
}
```

如果遇到格式不符合的情況：
1. **優先做法**：聯繫後端團隊調整 API 回應格式，統一使用標準格式
2. **臨時方案**（不推薦，會增加維護成本）：

```javascript
// 臨時方案範例（不推薦）
const { data: apiResponse } = useGetDataList(apiParams)

const dataList = computed(() => {
  if (!apiResponse.value) return []
  
  // 將非標準格式轉換為標準格式
  const standardFormat = {
    content: apiResponse.value.items,
    page: apiResponse.value.currentPage,
    size: apiResponse.value.perPage,
    totalPages: apiResponse.value.pages,
    total: apiResponse.value.totalItems
  }
  
  return extractPaginationContent(standardFormat)
})
```

> 💡 發現格式不符時，請回報給後端團隊統一調整，避免每個頁面都要寫轉換邏輯。

#### Q7: 如何在頁碼變更時觸發額外動作？
使用 `watchEffect` 或 `watch`：

```javascript
import { useAppScroll } from '@/composables/useAppScroll'

const { scrollToTop } = useAppScroll()

watchEffect(() => {
  // 當 page 改變時，捲動到頂部
  if (paginationModel.value.page) {
    scrollToTop()
    console.log('切換到第', paginationModel.value.page, '頁')
  }
})
```

#### Q8: AppTable 沒有 pagination prop 怎麼辦？
這種情況表示這個 app-table 只用來展示靜態資料，不需要分頁功能：

```vue
<!-- 靜態表格，只需要 accessible -->
<app-table accessible>
  <tbody>
    <tr>
      <td>資料1</td>
    </tr>
    <tr>
      <td>資料2</td>
    </tr>
  </tbody>
</app-table>
```

只有需要分頁功能的表格才需要加上 `:pagination` 和 `@update:page`。

#### Q9: 為什麼不能直接用 paginationModel.page = page 處理換頁？

因為 `mergePaginationParameter` 內部使用獨立的 `requestPage` 來產生 `pageNo`，而非讀取 `paginationModel.page`。

直接修改 `paginationModel.page` 只會更新 UI 顯示，**不會觸發 API 請求**。必須透過 `updatePaginationPage` 同時更新 `requestPage` 和 `paginationModel.page`：

```javascript
// ❌ 錯誤：只更新 UI，不觸發 API
paginationModel.value.page = 2

// ✅ 正確：同時更新 requestPage 和 UI，觸發 API
updatePaginationPage(2)
```

**設計原因：**

`extractPaginationContent` 會從 API 回應中回寫 `paginationModel.page`（同步總頁數、總筆數等 UI 狀態）。如果 `mergePaginationParameter` 直接讀取 `paginationModel.page`，就會形成無限循環：

```
extractPaginationContent 寫入 paginationModel.page
  → mergePaginationParameter 的 computed 重算
  → 產生新物件觸發 vue-query
  → API 請求 → 回應
  → extractPaginationContent 又寫入
  → 無限循環 ♻️
```

透過 `requestPage` 將「API 請求用的頁碼」和「UI 顯示用的頁碼」分離，徹底切斷循環。

## ⚙️ 後端實作指引

### 📌 統一規範

本專案統一採用以下兩項**強制規定**，其餘 Service / Repository 的實作方式不限制：
- **Request：** 含有分頁參數的 Request 物件，一律繼承 `DefaultPageParams`
- **Response：** 分頁查詢的回應，一律回傳 `Pagination<T>`

---

### 1. 核心類別說明

#### Request 端

| 類別 | 路徑 | 說明 |
|---|---|---|
| `PageParams` | moi.rdss.platform.paging.PageParams | 分頁參數介面，定義 getPageNo() / getPageSize() |
| `DefaultPageParams` | moi.rdss.platform.paging.DefaultPageParams | 分頁參數基礎類，預設 pageNo=1、pageSize=10 |

#### Response 端

| 類別 | 路徑 | 說明 |
|---|---|---|
| `Pagination<T>` | moi.rdss.platform.paging.Pagination | 統一分頁回應物件（自訂），所有分頁 API 均回傳此型別 |

#### 工具類（選用）

| 類別 | 方法 | 說明 |
|---|---|---|
| `RequestUtils` | getPageRequest(PageParams) | 將 Request 轉為 Spring 的 PageRequest，自動處理頁碼從 1 → 0 的轉換 |

---

### 2. Request：繼承 DefaultPageParams

> ✅ **強制規範：** 需要分頁的 Request 物件，一律繼承 `DefaultPageParams`，不可自行重複定義 pageNo / pageSize 欄位。

```java
@Data
@SuperBuilder
@NoArgsConstructor
public class FooListReq extends DefaultPageParams {

    @Schema(description = "名稱關鍵字")
    private String name;

    @Schema(description = "狀態")
    private String status;
}
```

**DefaultPageParams 已內建以下欄位：**

| 欄位 | 預設值 | 說明 |
|---|---|---|
| `pageNo` | 1 | 前端頁碼從 1 開始 |
| `pageSize` | 10 | 每頁預設 10 筆 |

---

### 3. Response：回傳 Pagination\<T\>

> ✅ **強制規範：** 所有分頁查詢的 API 回應，一律包裝為 `Pagination<T>`，不可直接回傳 Spring 的 `Page<T>` 給前端。

**Pagination\<T\> JSON 回應格式：**
```json
{
  "content": [],
  "total": 100,
  "size": 10,
  "number": 1,
  "totalPages": 10,
  "numberOfElements": 10,
  "first": true,
  "last": false,
  "offset": 0,
  "empty": false,
  "extra": {}
}
```

| 欄位 | 型別 | 說明 |
|---|---|---|
| `content` | Array | 資料集合 |
| `total` | Integer | 總筆數 |
| `size` | Integer | 每頁筆數 |
| `number` | Integer | 目前頁次（從 1 開始） |
| `totalPages` | Integer | 總頁數 |
| `numberOfElements` | Integer | 本頁實際筆數 |
| `first` | Boolean | 是否為第一頁 |
| `last` | Boolean | 是否為最後一頁 |
| `offset` | Integer | 分頁位移 |
| `empty` | Boolean | 是否有資料 |
| `extra` | Object | 額外附加資料（選用） |

---

### 4. Service / Repository 實作彈性說明

Service 與 Repository 的實作方式依 SQL 複雜程度自行決定，沒有強制標準寫法：

| 情境 | Repository 回傳 | Service 最終組裝 |
|---|---|---|
| 簡單查詢，條件單純 | Spring Data JPA 的 `Page<Entity>` | 轉換後包裝成 `Pagination<T>` |
| 複雜 SQL，需自組查詢 | `Page<Entity>` 或 `List<Entity>` 皆可 | 轉換後包裝成 `Pagination<T>` |

> **關鍵原則：** 不論 Service / Repository 內部怎麼實作，最終在 Controller 回傳給前端的格式一律是 `Pagination<T>`，確保前端收到的分頁格式固定一致。

---

### 5. Pagination\<T\> 常用建構方式

實作時依情境選擇適合的建構子，以下為常見用法供參考：

```java
// 搭配 Spring Data JPA 的 Page<Entity> 使用（最常見）
return new Pagination<>(dtos, pageable, page.getTotalElements());

// 手動指定分頁資訊（複雜 SQL / Native SQL）
return new Pagination<>(dtos, pageNumber, pageSize, total);

// 附加額外欄位（如統計數字、摘要）
Pagination<FooDto> result = new Pagination<>(dtos, pageable, total);
result.addExtra("totalAmount", 99999L);
return result;

// 從現有 Pagination 轉型
return existingPagination.map(entity -> convertToDto(entity));
```

> ⚠️ **注意：** Spring Data JPA 的頁碼從 0 開始，前端傳入的 pageNo 從 1 開始。可使用 `RequestUtils.getPageRequest(req)` 自動轉換，或自行處理 -1 的計算。

---

### 6. 禁止事項（Do NOT）

| ❌ 禁止 | ✅ 應改為 |
|---|---|
| 回傳 Spring 的 `Page<T>` 給前端 | 回傳 `Pagination<T>` |
| Request 物件自行重複定義 pageNo / pageSize 欄位 | 繼承 `DefaultPageParams` |

---

### 7. 快速對照表

| 層次 | Request 型別 | Response 型別 | 備註 |
|---|---|---|---|
| **Controller** | `XxxReq extends DefaultPageParams` | `ResponseEntity<Pagination<XxxDto>>` | 強制規範 |
| **Service** | 由 PG 自行決定 | `Pagination<XxxDto>` | 強制規範 |
| **Repository** | 由 PG 自行決定 | `Page<T>` 或 `List<T>` 皆可 | 依 SQL 複雜度決定 |
